# IntelliMate 用户体验优化设计方案

> 日期：2026-05-25
> 状态：已批准
> 范围：对话体验 + 交互增强 + 体验打磨
> 视角：日常小助手的使用者

## 概述

本文档描述 IntelliMate 前端用户体验的三阶段优化路线图。优化目标是让产品从"能用"提升为"好用"——可信赖、流畅、功能完整。

### 背景

当前核心问题（按影响程度排序）：

- 页面刷新/重连后消息全部丢失（仅存于前端内存）
- 每个 Agent 只有一个对话流，无法开新话题
- 不能重新生成 AI 回复
- 无 URL 路由（不能分享链接、不能用浏览器前进/后退）
- 流式输出完成时视觉跳跃
- 等待回复时输入框完全阻塞
- 缺少浏览器后台通知
- 新用户无引导，不知道能做什么

### 产品定位

日常小助手——用户每天打开 IntelliMate 和 Agent 对话、下达任务、查看执行结果。

### 约束

- 仅内部使用，不需要考虑多租户
- 后端已持久化消息（transcript_message 表），前端只缺加载逻辑
- 现有 Session 模型可复用，无需大规模 schema 重构

## Phase 1: 对话基础重建

**目标：** 让对话可信赖——刷新不丢、能开新话题、能重新生成、有 URL。

### 1.1 对话生命周期模型

#### 行为定义

- 每个 Agent 在 webchat 渠道始终有一个"活跃对话"
- 打开页面时，从后端加载当前活跃对话的历史消息
- 执行 `/clear` 命令时：
  1. 当前活跃对话标记为已归档
  2. 创建新的空白对话作为活跃对话
  3. 前端消息列表清空，开始新对话
- 已归档的对话可在"对话历史"中按时间线浏览

#### 数据模型变更

Session 表新增 `status` 字段：

| 值 | 含义 |
|----|------|
| `active` | 当前活跃对话（每个 Agent 在每个渠道仅一个） |
| `archived` | 已归档对话（通过 /clear 产生） |

归档后的 Session 保留所有 transcript_message 记录不变。

#### 对话标题生成

归档时自动生成标题：取该对话第一条用户消息的前 30 个字符作为标题，存储在 Session 表新增的 `title` 字段中。

### 1.2 消息持久化 & 历史加载

#### 新增 REST API

| 方法 | 路径 | 用途 |
|------|------|------|
| GET | `/api/sessions/{agentName}/messages?limit=50&before={id}` | 获取当前活跃对话的消息（支持向上翻页） |
| POST | `/api/sessions/{agentName}/clear` | 归档当前对话，创建新会话 |
| GET | `/api/sessions/{agentName}/archived` | 获取已归档对话列表 |
| GET | `/api/sessions/{sessionId}/messages?limit=100` | 查看某个已归档对话的消息 |

#### 消息响应格式

```json
{
  "messages": [
    {
      "id": 12345,
      "role": "user",
      "content": "帮我总结一下今天的工作",
      "createdAt": "2026-05-25T10:30:00",
      "toolCalls": null
    },
    {
      "id": 12346,
      "role": "assistant",
      "content": "好的，让我帮你总结...",
      "createdAt": "2026-05-25T10:30:05",
      "toolCalls": [
        { "name": "readFile", "arguments": "...", "result": "...", "success": true }
      ]
    }
  ],
  "hasMore": true
}
```

工具调用信息从 `metadataJson` 字段解析并随消息一起返回。

#### 前端加载时序

1. WebSocket `session.welcome` 事件到达
2. 前端调用 `GET /api/sessions/{agentName}/messages?limit=50`
3. 将返回的消息渲染到聊天界面（`streaming=false`）
4. 后续 WebSocket 实时消息追加到列表末尾

#### 向上翻页加载

用户滚动到消息列表顶部时，使用最早一条消息的 id 作为 `before` 参数请求更早的消息。

### 1.3 消息重新生成

#### 交互方式

- 最后一条助手消息的底部显示"重新生成"按钮（仅在非流式状态且是最新助手消息时显示）
- 点击后：
  1. 前端移除最后一条助手消息
  2. 重新发送 `conversation.message` 请求，附带 `regenerate: true` 标志
  3. 后端识别 regenerate 标志后，不在 transcript 中重复保存用户消息，直接生成新回复

#### 限制

- 只能重新生成最后一条回复（不支持中间消息的重新生成）
- 重新生成后旧回复不保留（不做版本管理）

### 1.4 URL 路由

#### 路由表

| 路由 | 页面 |
|------|------|
| `/` | 重定向到 `/chat` |
| `/chat` | 当前活跃 Agent 的对话 |
| `/chat/history` | 对话历史列表 |
| `/chat/history/:sessionId` | 查看某个归档对话（只读） |
| `/agents` | Agent 配置管理 |
| `/tools` | 工具管理 |
| `/skills` | Skills 管理 |
| `/models` | 模型管理 |
| `/memory` | 记忆观测 |
| `/scheduler` | 调度中心 |

#### 技术选型

引入 `react-router-dom` v7，替换现有的 `ViewMode` 状态驱动。

#### 迁移策略

- 侧边栏导航项从 `onClick` + `setViewMode` 改为 `<Link to="...">` 或 `navigate()`
- App.tsx 的条件渲染替换为 `<Routes>` + `<Route>` 结构
- 浏览器前进/后退自然生效

## Phase 2: 交互增强

**目标：** 让日常使用体验流畅、不阻塞、有感知。

### 2.1 流式渲染平滑化

#### 问题

流式输出时用 `<pre>` 纯文本渲染，完成后切换到 react-markdown 渲染，导致：
- 字体/字号变化
- 行高/段间距重排
- 代码块突然出现边框和高亮

#### 方案

增量 Markdown 渲染策略：
- 对已接收的内容按段落（双换行分隔）划分
- 已完成的段落（后面还有更多内容到达的）做 Markdown 渲染
- 最后一个不完整的段落保持纯文本 + 光标
- 完成时只有最后一段发生渲染切换，视觉跳跃极小

### 2.2 等待时可输入（排队机制）

#### 问题

`isWaiting=true` 时 textarea 被 disabled，用户思路被打断。

#### 方案

- 等待回复时，输入框保持可编辑
- 发送按钮变为"排队"图标，提示"回复完成后自动发送"
- 消息进入本地发送队列（最多 1 条排队消息）
- 当前回复完成（`agent.done`）后自动从队列取出发送
- 如果用户在等待期间点取消，排队的消息不发送

### 2.3 浏览器通知

#### 触发条件

- 页面不可见（`document.hidden === true`）
- 收到 `agent.done` 事件（回复完成）

#### 通知内容

- 标题：`[Agent名称] 回复了你`
- 正文：回复内容前 50 字
- 点击通知：回到页面并滚动到最新消息

#### 权限处理

- 首次收到后台回复时请求 `Notification.requestPermission()`
- 如果用户拒绝，不再请求

### 2.4 对话搜索

#### 入口

聊天界面顶部（TopBar 或消息列表上方）增加搜索图标。

#### 搜索范围

- 当前活跃对话的消息
- 所有已归档对话的消息（后端全文搜索）

#### 结果展示

- 搜索结果列表显示匹配的消息片段 + 所属对话标题 + 时间
- 点击结果：
  - 如果在当前对话中，跳转到对应消息位置
  - 如果在归档对话中，跳转到归档对话的只读视图

#### 新增 API

`GET /api/sessions/{agentName}/search?q={keyword}&limit=20`

### 2.5 侧边栏导航优化

#### 现状

7 个管理页面平铺显示，对日常对话用户是信息噪音。

#### 优化后结构

```
[当前 Agent 名称]          ← 点击弹出 Agent 切换列表
[+ 新对话]                 ← 执行 /clear 的快捷入口
[🔍 搜索]                 ← 对话搜索入口
[📋 对话历史]              ← 查看归档对话
─────────────────
▸ 管理                    ← 默认收起，展开后显示：
    Agent 配置
    工具管理
    Skills 管理
    模型管理
─────────────────
[📊 调度中心]
[🧠 记忆观测]
─────────────────
连接信息
```

将高频操作（新对话、搜索、历史）放在顶部，低频管理操作折叠收起。

## Phase 3: 体验打磨

**目标：** 功能完整性，让用户知道能做什么、能获取文件信息、能导出成果。

### 3.1 Onboarding / 空状态引导

#### 触发条件

当前对话为空（新建对话或首次使用）时显示。

#### 引导卡片内容

- Agent 介绍：来自 `goal` 字段或 `soul_md` 的第一段
- 推荐提问示例：3 个可点击的示例问题（点击直接发送）
- 功能提示：简要标注"/ 命令"、"计划模式"、"工具调用"等核心能力
- 可配置性：每个 Agent 可在配置中设定自己的推荐问题

#### 首次使用欢迎

首次打开应用时，叠加一个简短的 3 步引导 overlay：
1. "这是你的 AI 助手"
2. "试试输入任何问题"
3. "用 /plan 创建执行计划"

### 3.2 文件/图片上传

#### 交互方式

- 输入区域左侧增加附件按钮（📎）
- 支持点击选择文件 + 拖拽到输入区域
- 上传后在输入框上方显示文件预览缩略图

#### 支持类型

- 文本文件（.txt, .md, .json, .csv 等）→ 内容直接作为消息上下文
- 图片（.jpg, .png, .webp）→ 如模型支持多模态则直接传入，否则 OCR 提取文本
- PDF → 提取文本内容

#### 后端处理

- 新增文件上传接口 `POST /api/upload`
- 文件临时存储（本地磁盘或内存），返回可访问的 URL
- 消息发送时将文件 URL 作为上下文附加到请求中
- 可配置文件大小限制（默认 10MB）

### 3.3 Agent 能力展示

#### 入口

TopBar 中 Agent 名称旁增加信息图标（ℹ️），点击展开能力面板。

#### 展示内容

- 使用的模型名称
- 已启用的工具（按类别分组：内置工具 / MCP 工具 / 自定义工具）
- 已启用的 Skills 列表
- 是否支持计划模式
- 是否支持委派协作
- Bridge 连接状态（如有）

### 3.4 对话导出

#### 入口

聊天界面右上角或消息列表操作栏中的"导出"按钮。

#### 导出格式

- Markdown 文本（复制到剪贴板）
- .md 文件下载
- 可选：仅导出 AI 回复 / 完整对话

#### 导出范围

- 默认：当前对话全部消息
- 可选：手动选择消息范围

### 3.5 计划完成总结视图

#### 触发

计划状态变为 `completed` 时，在聊天消息流中自动插入总结卡片。

#### 卡片内容

- 计划标题
- 总步骤数 / 成功数 / 失败数 / 跳过数
- 总执行耗时
- 各步骤状态一览（紧凑列表：序号 + 标题 + ✓/✗）
- 可展开查看每步的详细结果

## 阶段依赖与执行顺序

```
Phase 1 (对话基础重建)
    │
    ├──▶ 1.1 对话生命周期 → 必须先行（后续都依赖多会话模型）
    ├──▶ 1.2 消息持久化 → 依赖 1.1 的 session 状态定义
    ├──▶ 1.3 消息重新生成 → 可独立开发，不依赖 1.1/1.2
    └──▶ 1.4 URL 路由 → 可独立开发，与 1.1-1.3 无硬依赖
    
Phase 2 (交互增强)
    │
    ├──▶ 2.1 流式渲染 → 纯前端，随时可做
    ├──▶ 2.2 排队机制 → 纯前端，随时可做
    ├──▶ 2.3 浏览器通知 → 纯前端，随时可做
    ├──▶ 2.4 对话搜索 → 依赖 Phase 1（有归档对话后搜索才有意义）
    └──▶ 2.5 侧边栏优化 → 依赖 1.4 路由（导航项需要路由链接）

Phase 3 (体验打磨)
    │
    └──▶ 均可独立开发，无硬依赖顺序
```

推荐执行顺序：1.1 → 1.2 → 1.4 → 1.3 → 2.1/2.2/2.3(并行) → 2.5 → 2.4 → Phase 3

## 风险与缓解

| 风险 | 缓解措施 |
|------|---------|
| 消息加载量大导致页面卡顿 | 分页加载（默认 50 条）+ 虚拟滚动 |
| Session 表 status 字段影响现有查询 | 默认值 active，现有逻辑加 WHERE status='active' 条件 |
| 流式增量 Markdown 渲染复杂度高 | 使用成熟方案（如按段落渲染），保留纯文本降级 |
| URL 路由引入后 PlanPanel 侧边栏状态丢失 | PlanPanel 状态保持在全局 store，路由切换不影响 |
| 文件上传安全风险 | 限制文件类型和大小，临时文件定时清理 |

## 验证标准

| 阶段 | 验证 |
|------|------|
| Phase 1 | 刷新页面后消息保留；/clear 后旧对话可在历史中查看；重新生成功能正常；URL 可分享和前进后退 |
| Phase 2 | 流式输出无明显跳跃；等待时可编辑输入；切换标签页后收到通知；搜索可找到历史对话内容 |
| Phase 3 | 空白对话显示引导；可上传文件并被 Agent 处理；可导出对话为 Markdown |
