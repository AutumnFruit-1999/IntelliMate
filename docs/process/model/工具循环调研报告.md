# 单 Agent 工具循环调研报告

> 日期: 2026-03-10  
> 状态: Final  
> 目标: 分析主流开源项目/框架在单 Agent 工具循环（ReAct Loop）中的设计，为 IntelliMate AgentRuntime 优化提供参考

---

## 1. 调研背景

### 1.1 什么是工具循环

单 Agent 工具循环（Tool Loop / ReAct Loop）是 LLM Agent 的核心执行机制：

```
User Message → LLM 推理 → 判断是否需要工具?
  ├── 需要 → 执行 Tool → 将结果送回 LLM → 继续推理（循环）
  └── 不需要 → 输出最终回复
```

这个循环决定了 Agent 的以下关键特性：
- **执行效率**: 工具是串行还是并行执行
- **稳定性**: 如何处理工具失败、无限循环、上下文溢出
- **可观测性**: 执行过程对用户是否可见、可干预
- **成本控制**: token 消耗是否可控、是否有预算限制
- **上下文质量**: 随着 turn 增多，上下文是否被有效管理

### 1.2 IntelliMate 现状

IntelliMate 的工具循环实现在 `AgentRuntime.java`，采用标准 ReAct 模式：

```
dispatch() → RunQueueManager.enqueue() → executeAgentLoop()
  └── executeLoopTurn()
        ├── LLM stream (收集 chunks)
        ├── 检查 tool calls
        │     └── processToolCalls()
        │           ├── concatMap 串行执行每个 tool call
        │           ├── 将结果作为 ToolResponseMessage 加入 history
        │           └── 递归调用 executeLoopTurn(turn + 1)
        └── 无 tool calls → Done
```

**现有配置**:
- `maxTurns`: 最大循环轮数（默认 128）
- `timeoutSeconds`: 单次 LLM 调用超时（默认 300s）
- `internalToolExecutionEnabled(false)`: 手动管理工具执行

## 2. 逐项分析

### 2.1 OpenHands (原 OpenDevin)

**仓库**: github.com/All-Hands-AI/OpenHands  
**语言**: Python  
**版本**: v1.5.0 (2026-03)

#### 工具循环架构

OpenHands 采用 **EventStream 驱动的 AgentController 循环**：

```
AgentController.run()
  └── while step < MAX_ITERATIONS:
        ├── agent.step(state) → Action
        ├── if Action is tool call:
        │     └── runtime.execute(action) → Observation
        │           ├── truncate_observation(obs, MAX_CHARS)
        │           └── append to EventStream
        ├── if stuck_detector.is_stuck():
        │     └── inject unstuck hint / raise StuckError
        └── if Action is AgentFinishAction:
              └── break
```

#### 核心机制详解

**1. 工具结果截断 (MAX_CHARS)**

```python
MAX_CHARS = 10_000  # 可配置

def truncate_observation(observation, max_chars=MAX_CHARS):
    if len(observation.content) > max_chars:
        half = max_chars // 2
        observation.content = (
            observation.content[:half]
            + "\n... [truncated, showing first and last {} chars] ...\n"
            + observation.content[-half:]
        )
```

设计要点：
- 截断保留**头尾各一半**，而非只保留开头（尾部常有关键结果/错误信息）
- 字符级截断（非 token 级），因为 tokenization 发生在 LLM 侧
- 截断标记明确告知 LLM 内容被省略，LLM 可据此决定是否重新读取

**2. Condenser 上下文压缩系统**

OpenHands 的亮点设计——多策略可插拔的上下文压缩：

```
CondenserBase (抽象接口)
  ├── NoOpCondenser        — 不压缩（调试用）
  ├── RollingCondenser      — 滑动窗口，保留最近 N 条事件
  ├── LLMSummarizingCondenser — 使用 LLM 将旧历史压缩为摘要
  └── PipelineCondenser     — 多策略链式组合
```

LLMSummarizingCondenser 的工作方式：
- 当 event history 超过阈值时触发
- 使用单独的 LLM 调用将旧历史压缩为结构化摘要
- 摘要重点保留：用户目标、已完成步骤、关键文件路径、失败的测试
- 压缩后上下文管理从**二次增长**降为**线性增长**
- 每次压缩事件被记录为 `CondensationAction`，支持状态回溯

**3. Stuck 检测与恢复**

```python
class StuckDetector:
    def is_stuck(self, history):
        # 检测重复的 action pattern
        recent = history[-5:]
        if all_same_action(recent):
            return True
        # 检测空循环（LLM 反复输出无效 action）
        if count_noop(recent) > 3:
            return True
        return False
```

检测到 stuck 后的处理策略：
1. 注入提示 ("You seem stuck, try a different approach")
2. 如果仍然 stuck，抛出 `StuckError` 终止循环
3. 循环检测 + condenser 配合：condenser 自身也有循环检测（防止压缩→扩展→压缩的死循环）

**4. 资源限制**

| 限制 | 默认值 | 可配置 | 目的 |
|------|--------|--------|------|
| MAX_ITERATIONS | 100 | 是 | 防止无限循环 |
| MAX_CHARS | 10,000 | 是 | 防止单条 observation 过大 |
| MAX_BUDGET_PER_TASK | 无限 | 是 | 成本控制 |

#### 对 IntelliMate 的启示

- **必须实现工具结果截断**，头尾保留策略优于只保留开头
- **Condenser 系统**是长对话场景的必备能力，LLMSummarizingCondenser 效果最好但成本高
- **Stuck 检测**可以用简单的重复 pattern 匹配实现，投入产出比高

### 2.2 DeerFlow 2.0 (ByteDance)

**仓库**: github.com/bytedance/deer-flow  
**语言**: Python  
**基于**: LangGraph + LangChain

#### 工具循环架构

DeerFlow 基于 LangGraph 的 StateGraph，工具循环嵌套在 Agent Node 中：

```
StateGraph:
  researcher_node:
    └── ReAct Agent Loop (LangGraph 内置)
          ├── LLM → tool_calls?
          │     └── ToolNode.execute(tool_calls)
          │           ├── 沙箱执行（代码工具）
          │           ├── API 调用（搜索/爬虫）
          │           └── 结果作为 ToolMessage 返回
          └── LLM → final answer → 退出 Agent Node
```

#### 核心机制详解

**1. LangGraph ToolNode 标准执行**

DeerFlow 使用 LangGraph 的 `ToolNode`，它提供了标准化的工具执行管道：

```python
class ToolNode:
    def __init__(self, tools, handle_tool_errors=True):
        self.tools = tools
        self.handle_tool_errors = handle_tool_errors
    
    def __call__(self, state):
        tool_calls = state["messages"][-1].tool_calls
        results = []
        for tc in tool_calls:
            try:
                result = self.tools[tc["name"]].invoke(tc["args"])
                results.append(ToolMessage(content=result, tool_call_id=tc["id"]))
            except Exception as e:
                if self.handle_tool_errors:
                    results.append(ToolMessage(
                        content=f"Error: {e}", tool_call_id=tc["id"]))
                else:
                    raise
        return {"messages": results}
```

`handle_tool_errors` 支持三种模式：
- `True` — 捕获所有异常，包装为 ToolMessage 返回给 LLM
- `str` — 使用固定的错误消息模板
- `callable` — 自定义错误处理函数

**2. 工具循环检测 (LoopDetectionMiddleware)**

DeerFlow 遇到的一个关键问题：Agent 在特定模型（如 DeepSeek、Qwen）下会陷入**重复调用相同工具相同参数**的死循环：

```python
class LoopDetectionMiddleware:
    def __init__(self, window_size=5, warn_threshold=3, terminate_threshold=5):
        self.recent_calls = deque(maxlen=window_size)
    
    def check(self, tool_call):
        call_signature = (tool_call.name, hash(tool_call.args))
        self.recent_calls.append(call_signature)
        
        repeated = sum(1 for c in self.recent_calls if c == call_signature)
        
        if repeated >= self.terminate_threshold:
            # 强制终止：剥离 tool_calls，让 LLM 生成文本回复
            return "terminate"
        elif repeated >= self.warn_threshold:
            # 警告：注入提示
            return "warn"
        return "ok"
```

策略亮点：
- 滑动窗口 + 签名匹配，低成本检测
- **渐进式响应**：先警告再终止，给 LLM 自我修正的机会
- 终止后不硬报错，而是剥离 tool_calls 让 LLM 输出现有结果（**优雅降级**）

**3. 沙箱执行隔离**

```
代码执行工具:
  ├── Python REPL → Docker sandbox
  ├── Bash 命令  → Docker sandbox
  └── 文件操作   → 受限文件系统

搜索/爬虫工具:
  ├── Tavily Search API
  ├── Web Crawler
  └── 直接 HTTP 调用
```

沙箱的目的：
- 代码执行结果可能很长（数据分析输出），沙箱内预先截断
- 超时保护（命令执行有独立超时）
- 安全隔离（防止恶意代码影响主机）

**4. recursion_limit 配置**

LangGraph 层面的递归深度限制：

```python
graph = create_react_agent(
    model=model,
    tools=tools,
    state_modifier=system_prompt
)
result = graph.invoke(
    {"messages": [HumanMessage(content=query)]},
    config={"recursion_limit": 25}
)
```

- 默认 25 次递归（约等于 12-13 个 tool call 轮次，因为每次 LLM→Tool→LLM 算 2 次递归）
- 超出后抛出 `GraphRecursionError`
- DeerFlow 将此设为可配置参数

#### 对 IntelliMate 的启示

- **工具循环检测是必须的**，特别是使用非 OpenAI 模型时
- **渐进式降级**优于硬停止：先警告再终止，给 LLM 自修正机会
- 沙箱执行是 exec 类工具的最佳实践（当前 IntelliMate 的 ExecTool 直接在主机执行）

### 2.3 Cursor 2.0

**来源**: cursor.com/blog, cursor.com/docs  
**定位**: AI 代码编辑器（闭源）

#### 工具循环架构

Cursor 的单 Agent 工具循环是其多 Agent 体系的基础单元：

```
Single Agent Turn:
  ├── LLM 推理 (250 tokens/s, 极速)
  ├── tool_calls (最多 25 个/turn)
  │     ├── 并行执行（无依赖的 tool calls 同时运行）
  │     └── 结果流式返回（不等所有工具完成）
  ├── 结果送回 LLM
  └── 继续推理或输出
```

#### 核心机制详解

**1. 并行工具执行**

Cursor 最显著的设计决策：**同一 turn 的多个 tool calls 并行执行**。

```
LLM 返回 3 个 tool calls:
  tool_call_1: readFile("src/a.java")  ─┐
  tool_call_2: readFile("src/b.java")  ─┼─→ 并行执行
  tool_call_3: searchFiles("pattern")  ─┘
                                          │
  收集所有结果 ←──────────────────────────┘
  └── 组装为 ToolResponseMessage[] 送回 LLM
```

并行执行的前提和约束：
- 同一 turn 的 tool calls 被视为**无依赖**（LLM 在生成时已隐式保证）
- 最多 25 个/turn（防止过度并行）
- 写操作需要更谨慎（文件写冲突处理）

**2. Token Budget 管理**

Cursor 对 token 消耗有精细管控：

| 维度 | 策略 |
|------|------|
| **模型选择** | 快速模型用于简单工具（Explore），强模型用于核心推理 |
| **上下文裁剪** | 只附加必要文件，避免发送完整代码库 |
| **结果缓存** | 相同 readFile 请求在同一 session 内缓存 |
| **增量传输** | 文件编辑只传 diff，不传完整文件 |

**3. Subagent 工具调用隔离**

Cursor 的 subagent 设计值得关注：

```
Main Agent (Agent Mode)
  ├── tool_call: explore("Where is auth implemented?")
  │     └── Explore Subagent (独立上下文, 快速模型)
  │           ├── searchFiles, readFile, ...
  │           └── 返回摘要结果
  ├── tool_call: bash("npm test")
  │     └── Shell Subagent (独立上下文)
  │           └── 返回命令输出
  └── 继续推理
```

subagent 的优势：
- **上下文隔离**: subagent 有独立的上下文，不污染主 Agent 的 history
- **模型分级**: subagent 可以用更便宜/更快的模型
- **结果压缩**: subagent 只返回摘要，不返回完整的中间过程

**4. Streaming Tool Results**

Cursor 支持工具执行过程中的流式反馈：

- 命令执行（bash）时实时流式输出
- 文件搜索时逐步返回匹配结果
- 用户可以在工具执行过程中看到进度

#### 对 IntelliMate 的启示

- **并行工具执行是最直接的性能提升**，IntelliMate 的 `concatMap` 应改为 `flatMap`/`merge`
- **token budget 管理**需要建立：至少实现结果截断和总量追踪
- **subagent 模式**可以用于隔离重工具（如 exec）的上下文
- 25 个 tool calls/turn 的限制值得参考

### 2.4 LangGraph

**仓库**: github.com/langchain-ai/langgraph  
**定位**: Agent 编排框架

#### 工具循环架构

LangGraph 提供了工具循环的**标准化框架**：

```python
# create_react_agent 是 LangGraph 内置的 ReAct 循环
agent = create_react_agent(
    model=model,
    tools=tools,
    checkpointer=MemorySaver(),  # 检查点
    interrupt_before=["tools"],   # 工具执行前中断（人工审批）
)
```

内部实现等效于：

```
StateGraph:
  agent_node:
    ├── 调用 LLM
    ├── 条件边: has_tool_calls?
    │     ├── 是 → tools_node
    │     └── 否 → END
    └── tools_node:
          ├── ToolNode 执行所有 tool calls
          └── 边 → agent_node (回到 LLM)
```

#### 核心机制详解

**1. ToolNode 错误处理**

LangGraph 的 `ToolNode` 支持多种错误处理策略：

```python
ToolNode(
    tools=[...],
    handle_tool_errors=True,          # bool: 捕获所有异常
    # handle_tool_errors="重试提示",    # str: 固定错误消息
    # handle_tool_errors=my_handler,   # callable: 自定义处理
    # handle_tool_errors=(ValueError,) # tuple: 只处理特定异常
)
```

错误被包装为 `ToolMessage` 返回给 LLM，LLM 可以：
- 修改参数重试
- 换一个工具
- 直接用已有信息回答

这比硬抛异常终止循环优雅得多。

**2. Human-in-the-Loop 中断**

LangGraph 的人工介入设计是所有框架中最成熟的：

```python
# 配置：在 tools 节点执行前中断
agent = create_react_agent(
    model=model,
    tools=tools,
    interrupt_before=["tools"]
)

# 执行到 tools 节点前暂停
state = agent.invoke({"messages": [HumanMessage("删除数据库")]})
# state 被持久化到 checkpointer

# 人工审查 pending tool calls
pending_tools = state["messages"][-1].tool_calls
# 用户决定: 批准 / 修改 / 拒绝

# 批准后继续
agent.invoke(None, config={"thread_id": thread_id})

# 或者注入修改后的 tool call
agent.invoke(
    Command(update={"messages": [modified_tool_message]}),
    config={"thread_id": thread_id}
)
```

**3. Checkpoint 恢复**

```python
# 自动保存每个节点执行后的状态
checkpointer = MemorySaver()  # 或 SqliteSaver, PostgresSaver

# 任意时刻恢复
state = agent.get_state(config={"thread_id": "abc"})
# 回滚到特定 checkpoint
agent.update_state(config, values=previous_state)
```

Checkpoint 的价值：
- 工具执行失败后可以回滚到上一个状态
- 支持 "time travel" 调试
- 长时间运行的任务可以中断后恢复

**4. recursion_limit + 超时**

```python
config = {
    "recursion_limit": 25,        # 最大递归深度
    "max_concurrency": 5,          # 最大并行度
}
```

#### 对 IntelliMate 的启示

- **人工审批节点**（interrupt_before）的设计非常干净，可以通过 WebSocket 事件实现
- **Checkpoint**的概念可以用于实现"回滚到上一步"
- **错误处理策略**应该多样化，而非一刀切

### 2.5 Spring AI（当前框架）

**仓库**: github.com/spring-projects/spring-ai  
**版本**: 1.1.x

#### 框架原生能力

Spring AI 提供了两种工具执行模式：

```java
ToolCallingChatOptions.builder()
    .toolCallbacks(tools)
    // 模式 1: 框架自动执行工具循环
    .internalToolExecutionEnabled(true)
    // 模式 2: 手动管理工具循环 (IntelliMate 当前选择)
    .internalToolExecutionEnabled(false)
    .build();
```

**模式 1 (internalToolExecutionEnabled=true)**:

```
ChatModel.call(prompt) 内部:
  while (response.hasToolCalls()):
      execute all tool callbacks
      append results to messages
      call LLM again
  return final response
```

- 优点：零代码实现工具循环
- 缺点：无法流式输出中间过程、无法自定义执行策略、无法拦截工具调用

**模式 2 (internalToolExecutionEnabled=false, IntelliMate 当前)**:

- 完全手动管理循环
- 可以流式输出每个 chunk 和工具事件
- 可以自定义工具执行逻辑
- 代价：需要自己处理所有边界情况

#### 框架限制

| 限制 | 描述 | 影响 |
|------|------|------|
| **无 token 计数 API** | ChatResponse 中有 `usage` 字段但并非所有 provider 都填充 | 无法精确追踪消耗 |
| **无内置截断** | 框架不对 tool result 做任何处理 | 大结果直接膨胀 history |
| **无循环保护** | `internalToolExecutionMaxIterations` 仍在 PR 中（#3380），未合并 | 需要自行实现 maxTurns |
| **streaming 不含 tool 执行** | stream 只流式 LLM 的 token，工具执行是同步阻塞 | 前端看不到工具执行进度 |

#### 对 IntelliMate 的启示

- IntelliMate 选择 `internalToolExecutionEnabled(false)` 是正确的——保留了完全的控制权
- 但需要自行补齐框架未提供的能力：截断、循环保护、token 追踪
- `ChatResponse.getMetadata().getUsage()` 可用于 token 统计（需确认 provider 支持）

## 3. 横向对比

### 3.1 工具执行模型

| 项目 | 执行模型 | 每 turn 限制 | 执行线程 |
|------|----------|-------------|----------|
| **OpenHands** | 串行（每 turn 1 个 action） | 1 | 单线程 |
| **DeerFlow** | 串行（ToolNode 内遍历） | 无限制 | 单线程 |
| **Cursor** | **并行** | 25 | 多线程/协程 |
| **LangGraph** | 串行（ToolNode 遍历） | 无限制 | 支持 max_concurrency |
| **Spring AI** | 串行（手动循环） | 无限制 | 取决于实现 |
| **IntelliMate** | **串行 (concatMap)** | maxTurns=128 | boundedElastic |

### 3.2 工具结果截断

| 项目 | 截断策略 | 默认限制 | 截断方式 |
|------|----------|----------|----------|
| **OpenHands** | **头尾保留** | 10,000 chars | 中间用省略标记替换 |
| **DeerFlow** | 依赖 LangGraph | 无内置 | 工具自行控制 |
| **Cursor** | 智能裁剪 | 未公开 | diff/摘要 |
| **LangGraph** | **无内置** | 无 | 用户自行处理 |
| **Spring AI** | **无内置** | 无 | 用户自行处理 |
| **IntelliMate** | **无** | 无 | 原样传入 history |

### 3.3 上下文管理

| 项目 | 上下文压缩 | token 追踪 | 溢出保护 |
|------|-----------|-----------|---------|
| **OpenHands** | **Condenser 系统** (NoOp/Rolling/LLM 摘要/Pipeline) | MAX_CHARS (字符级) | MAX_CHARS + Condenser |
| **DeerFlow** | 无（依赖 recursion_limit） | 无 | recursion_limit |
| **Cursor** | **增量传输 + 结果缓存** | token budget | 智能裁剪 |
| **LangGraph** | 无内置 | 无 | recursion_limit |
| **Spring AI** | 无 | usage metadata | 无 |
| **IntelliMate** | **无** | **无** | 仅 maxTurns |

### 3.4 错误恢复

| 项目 | 工具失败处理 | 循环检测 | 恢复机制 |
|------|-------------|---------|---------|
| **OpenHands** | 错误作为 Observation | **StuckDetector** | 注入提示 → 终止 |

## 4. 关键设计模式总结

### 模式一：工具结果截断

**目的**: 防止大文件/长输出撑爆 context window，降低 token 成本。

**最佳实践** (OpenHands):
```
原始结果 (50,000 chars)
  → 截断为 10,000 chars
  → 保留前 5,000 + "[truncated]" + 后 5,000
  → LLM 知道内容被截断，可以请求分段读取
```

**为什么头尾保留**:
- 文件开头有 import、类定义等结构信息
- 文件结尾有 return 语句、错误信息
- 中间部分信息密度通常较低

### 模式二：循环检测与优雅降级

**目的**: 防止 Agent 陷入死循环重复调用同一工具，浪费 token。

**最佳实践** (DeerFlow):
```
检测: 滑动窗口 + 调用签名匹配
  → 3 次重复: 警告 (注入提示)
  → 5 次重复: 终止 (剥离 tool_calls, 让 LLM 输出现有结果)
```

**为什么不硬停止**:
- 硬停止会丢失 LLM 已经积累的部分结果
- 剥离 tool_calls 后 LLM 被迫用已有信息生成回复（优雅降级）

### 模式三：并行工具执行

**目的**: 提升执行速度，特别是 I/O 密集型工具（文件读取、网络请求）。

**最佳实践** (Cursor):
```
LLM 返回的同一 turn 的多个 tool calls
  → 假设无依赖（LLM 保证）
  → 并行执行
  → 收集所有结果后一起返回
```

**限制**: 写操作需要特殊处理（文件写冲突），最大并行度需限制。

### 模式四：人工审批节点

**目的**: 高危操作（删除文件、执行命令、修改数据库）前让用户确认。

**最佳实践** (LangGraph):
```
配置: interrupt_before=["tools"]
  → 工具执行前暂停
  → 将 pending tool calls 发送给用户
  → 用户审批后继续
  → 状态通过 Checkpoint 持久化
```

### 模式五：上下文压缩

**目的**: 长对话中控制 token 消耗，保持 LLM 推理质量。

**最佳实践** (OpenHands Condenser):
```
策略选择:
  轻量级 → RollingCondenser (滑动窗口, 保留最近 N 条)
  高质量 → LLMSummarizingCondenser (使用 LLM 压缩旧历史)
  混合   → PipelineCondenser (先滑动窗口, 超出后 LLM 摘要)
```

## 5. 结论与推荐

### 5.1 IntelliMate 当前的差距

按严重度排序：

| 差距 | 严重度 | 影响 |
|------|--------|------|
| 无工具结果截断 | **严重** | 读取大文件直接撑爆 context，LLM 生成质量下降甚至报错 |
| 无循环检测 | **严重** | Agent 可能无限循环消耗 token（特别是 Qwen 模型） |
| 无 token 追踪 | **高** | 无法预测何时会超出 context window |
| 工具串行执行 | **中** | 多文件读取时性能低（I/O 等待叠加） |
| 中间过程不持久化 | **中** | 无法审计 Agent 的推理过程，无法从中间状态恢复 |
| 无人工审批 | **中** | exec/writeFile 等高危工具无拦截机制 |
| 无上下文压缩 | **低-中** | 长对话质量下降，但当前 maxTurns=128 足够大 |

### 5.2 推荐优化路径

```
P0 (安全与稳定):
  ├── 工具结果截断 (参考 OpenHands MAX_CHARS)
  ├── 循环检测 (参考 DeerFlow LoopDetectionMiddleware)
  └── Token 追踪 + 上下文窗口保护

P1 (性能与体验):
  ├── 并行工具执行 (参考 Cursor)
  ├── Tool call/result 持久化
  └── 工具重试策略 (参考 LangGraph handle_tool_errors)

P2 (高级能力):
  ├── 上下文压缩 (参考 OpenHands Condenser)
  ├── 人工审批节点 (参考 LangGraph interrupt_before)
  └── 工具结果缓存
```

详细优化方案见配套文档《AgentRuntime 工具循环优化方案.md》。
