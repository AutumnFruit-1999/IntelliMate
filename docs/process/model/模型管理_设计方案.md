# JavaClaw 模型管理 — 设计方案

> 版本：v1.0 | 日期：2026-03-11

---

## 1. 现状分析

### 1.1 当前模型配置

JavaClaw 目前仅接入阿里 **DashScope**（通义千问系列），模型配置存在以下问题：

| 问题 | 说明 |
|------|------|
| 单一厂商 | 仅 DashScope，无法使用 OpenAI、Anthropic、DeepSeek 等厂商的模型 |
| 全局唯一模型 | `ChatModel` bean 由 Spring AI DashScope 自动创建，所有 Agent 共享同一模型实例 |
| Agent 模型配置不生效 | `AgentEntity.model` 字段存储了模型名，但 `AgentRuntime` 构建 `Prompt` 时未将其传入 `ChatOptions`，实际调用始终使用 `application.yml` 中的全局模型 |
| 前端模型列表硬编码 | `AgentConfigModal.tsx` 中 `MODELS = ["qwen-plus", "qwen-max", "qwen-turbo", "qwen-long"]` 写死 |
| 无法编辑模型 | Agent 创建后无法在配置页面修改模型，仅创建时可选 |
| 默认值不一致 | `spring.ai.dashscope.chat.options.model` 默认 `qwen-plus`，`javaclaw.agent.model` 默认 `qwen-max` |

### 1.2 现有技术栈

- **Spring AI 1.0.0** + **Spring AI Alibaba 1.0.0.2**
- 自动配置：`DashScopeChatAutoConfiguration` → 注入单一 `ChatModel` bean
- Agent 模型字段流转：`AgentEntity.model` → `AgentConfigService.toAgentConfig()` → `ResolvedAgentConfig.agent.model` → `AgentRunRequest.agent.getModel()` → **未使用**

### 1.3 现有代码路径

```
application.yml                    AgentEntity (DB)
  spring.ai.dashscope.             ┌─ model: "qwen-max"
    chat.options.model: qwen-plus  └─ (已存储但未生效)
         │                                    │
         ▼                                    ▼
   DashScopeChatModel (全局 Bean)    AgentConfigService.resolve()
         │                                    │
         ▼                                    ▼
   AgentRuntime                      ResolvedAgentConfig.agent
     chatModel.stream(prompt)          .getModel() → 仅打日志
         │
         ▼
   所有 Agent 使用同一模型
```

## 2. 目标能力

### 2.1 整体目标

将模型系统从 **"单厂商、全局唯一"** 升级为 **"多厂商、动态注册、per-Agent 路由"**。

### 2.2 核心能力

| 能力 | 说明 |
|------|------|
| 多厂商支持 | 同时接入 DashScope、OpenAI（含兼容接口）、Anthropic 等 |
| 厂商管理 | 在前端页面添加/编辑/删除厂商配置（名称、类型、API地址、API Key） |
| 模型管理 | 每个厂商下维护可用模型列表，支持 CRUD |
| per-Agent 模型选择 | Agent 配置中从动态模型列表选择模型，不同 Agent 可用不同厂商的不同模型 |
| 动态路由 | 运行时根据 Agent 绑定的模型，动态路由到对应厂商的 ChatModel 实例 |
| API Key 安全 | 加密存储、前端脱敏展示 |
| 热更新 | 厂商/模型配置变更后无需重启 |

### 2.3 目标架构概览

## 3. 核心概念

### 3.1 概念模型

| 概念 | 英文 | 说明 | 示例 |
|------|------|------|------|
| 厂商 | Model Provider | AI 模型服务提供商，拥有 API 地址和认证凭据 | 阿里 DashScope、OpenAI、Anthropic、DeepSeek |
| 厂商类型 | Provider Type | 技术协议分类，决定使用哪种 ChatModel 实现 | `DASHSCOPE`、`OPENAI_COMPATIBLE`、`ANTHROPIC` |
| 模型定义 | Model Definition | 厂商下的具体模型，包含模型 ID 和能力描述 | qwen-plus、gpt-4o、claude-3.5-sonnet |
| 模型标识 | Model Key | 唯一定位一个模型的复合标识 | `provider:{providerId}:model:{modelId}` |
| Agent 绑定 | Agent Model Binding | Agent 与模型定义的关联关系 | Agent "javaclaw" 使用 "qwen-max" |

### 3.2 厂商类型定义

| 类型 | 协议 | 使用的 ChatModel 实现 | 典型厂商 |
|------|------|---------------------|---------|
| `DASHSCOPE` | 阿里 DashScope 原生协议 | `DashScopeChatModel` | 阿里通义千问 |
| `OPENAI_COMPATIBLE` | OpenAI Chat Completions API | `OpenAiChatModel`（可配自定义 baseUrl） | OpenAI、DeepSeek、Moonshot、Groq、零一万物、SiliconFlow |
| `ANTHROPIC` | Anthropic Messages API | `AnthropicChatModel` | Anthropic Claude |

> **OpenAI Compatible** 是最灵活的类型：大量国内外厂商提供 OpenAI 兼容接口，只需修改 `baseUrl` 即可接入。

### 3.3 关系图

```
model_provider (厂商)
  │ 1
  │
  │ N
  ▼
model_definition (模型)
  │ 1
  │
  │ N
  ▼
agent (智能体)
  └── model_id → 指向 model_definition.id
```

## 4. 数据模型

### 4.1 model_provider（厂商表）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | BIGINT PK | 自增主键 |
| name | VARCHAR(128) UNIQUE | 厂商显示名称，如"阿里 DashScope"、"OpenAI" |
| type | VARCHAR(32) NOT NULL | 厂商类型：`DASHSCOPE` / `OPENAI_COMPATIBLE` / `ANTHROPIC` |
| base_url | VARCHAR(512) NULL | API 基础地址（DashScope 可留空使用默认值） |
| api_key_encrypted | TEXT NOT NULL | AES 加密后的 API Key |
| enabled | TINYINT DEFAULT 1 | 是否启用 |
| sort_order | INT DEFAULT 0 | 排序权重 |
| created_at | DATETIME | 创建时间 |
| updated_at | DATETIME | 更新时间 |

### 4.2 model_definition（模型定义表）

| 字段 | 类型 | 说明 |
|------|------|------|
| id | BIGINT PK | 自增主键 |
| provider_id | BIGINT FK | 关联厂商 |
| model_id | VARCHAR(128) NOT NULL | 模型标识符（传给 API 的值），如 `gpt-4o`、`qwen-plus` |
| display_name | VARCHAR(128) NOT NULL | 前端显示名称，如"GPT-4o"、"通义千问 Plus" |
| description | TEXT NULL | 模型描述，帮助用户理解能力差异 |
| max_tokens | INT NULL | 最大输出 token 数（部分厂商需要设置） |
| capabilities | JSON NULL | 能力标签，如 `{"vision": false, "function_calling": true}` |
| enabled | TINYINT DEFAULT 1 | 是否启用 |
| sort_order | INT DEFAULT 0 | 排序权重 |
| created_at | DATETIME | 创建时间 |
| updated_at | DATETIME | 更新时间 |

> **UNIQUE 约束**：`(provider_id, model_id)` 联合唯一

### 4.3 agent 表改造

当前 `agent.model` 存储模型名称字符串（如 `qwen-max`），改造为存储 `model_definition.id`：

| 字段 | 旧值 | 新值 |
|------|------|------|
| model | `"qwen-max"` (字符串) | `"42"` (model_definition.id) 或保持字符串兼容 |

**兼容策略**：`model` 字段改为存储格式 `"provider:{providerId}:model:{modelId}"` 或直接存 `model_definition.id`。为平滑迁移，支持两种格式：
- 纯数字 → `model_definition.id`
- 非数字字符串 → 旧格式，回退到默认 DashScope provider 查找

|------|------|------|

| DELETE | `/api/model-providers/{id}` | 删除厂商（级联删除下属模型） |
| POST | `/api/model-providers/{id}/test` | 测试厂商连通性 |

## 8. 安全考虑

### 8.1 API Key 管理

| 措施 | 说明 |
|------|------|
| 加密存储 | 数据库中 `api_key_encrypted` 使用 AES-256-GCM 对称加密，密钥从环境变量 `JAVACLAW_CRYPTO_KEY` 读取 |
| 前端脱敏 | API Key 在 GET 响应中返回 `apiKeyMasked`（如 `sk-****b599`），不返回原文 |
| 更新保护 | PUT 请求中 `apiKey` 为空或 `null` 时不覆盖已有值 |
| 传输安全 | 所有 API 走 HTTPS（生产环境） |

### 8.2 前端脱敏规则

```
原始值: sk-8b71fb8815884080b52bd3b670b3b599
脱敏值: sk-****b599

规则: 保留前 3 字符 + "****" + 最后 4 字符
长度 < 8 时: 全部替换为 "****"
```

### 8.3 权限控制（未来扩展）

当前 JavaClaw 无用户权限体系。未来可增加：
- 仅管理员可操作厂商和模型
- 普通用户仅能在 Agent 配置中选择模型

## 9. 兼容性与迁移

### 9.1 向后兼容

- **保留 application.yml 配置**：`spring.ai.dashscope` 配置保留，作为系统启动时的默认 DashScope provider
- **自动迁移**：首次启动时，若 `model_provider` 表为空，自动从 `application.yml` 读取 DashScope 配置创建默认厂商和模型记录
- **Agent model 字段兼容**：旧数据中 `agent.model` 为字符串（如 `qwen-max`），迁移脚本将其映射为对应的 `model_definition.id`

### 9.2 迁移步骤

1. 执行 Flyway 迁移，创建 `model_provider` 和 `model_definition` 表
2. 数据迁移脚本：从 `application.yml` 导入默认 DashScope 厂商 + 常用 qwen 模型
3. 更新 `agent.model` 字段值（字符串 → model_definition.id）
4. 前端部署新版本

### 9.3 回退方案

- 若新版本有问题，`AgentRuntime` 保留 fallback：当 `ChatModelRegistry` 找不到指定模型时，回退到默认 DashScope ChatModel
- `agent.model` 字段保持 TEXT 类型，新旧格式可共存
